Case Study 1: Failed Login Investigation
Objective

The purpose of this investigation was to monitor failed Windows login attempts using Windows Security logs and Splunk. The goal was to understand how failed authentication events are recorded, identify the reason for the login failures, and practice investigating authentication events as a SOC analyst.

Lab Environment
Host Machine: Windows 10
SIEM: Splunk Enterprise
Log Forwarder: Splunk Universal Forwarder
Monitoring Tool: Windows Security Logs
Target System: Windows 10 Virtual Machine
Scenario

During this lab, I simulated multiple failed login attempts by locking my Windows 10 virtual machine and entering an incorrect password several times.

Each failed login generated Windows Security Event ID 4625, which was forwarded to Splunk using the Universal Forwarder.

My task was to investigate these events, determine why the logins failed, and identify the important details recorded by Windows.

Investigation Steps
Step 1 – Search for Failed Login Events

I first searched for all failed login events generated by Windows Security Logs.

SPL Query
index=* EventCode=4625

This displayed all failed login events collected during the investigation.

Step 2 – Count Failed Login Attempts

Next, I counted the total number of failed login events.

SPL Query
index=* EventCode=4625
| stats count as "Failed Login Attempts"

This confirmed that 11 failed login attempts were recorded during the lab.

Step 3 – Check the Failure Reason

To understand why the login attempts failed, I grouped the events by failure reason.

SPL Query
index=* EventCode=4625
| stats count by Failure_Reason

This showed that all login attempts failed because of an unknown user name or bad password.

Step 4 – Review the Logon Type

I then checked which logon type was used during the failed authentication attempts.

SPL Query
index=* EventCode=4625
| stats count by Logon_Type

The results showed Logon Type 2 (Interactive), indicating that the incorrect password was entered directly on the Windows login screen.

Step 5 – Display Important Event Details

To review the most useful information, I displayed the key fields from each event.

SPL Query
index=* EventCode=4625
| table _time Account_Name Account_Domain Failure_Reason Logon_Type Source_Network_Address Workstation_Name host

This allowed me to quickly identify:

The time of each failed login
The user account involved
The failure reason
The logon type
The source network address
The affected workstation
Step 6 – Create a Timeline

Finally, I created a timeline to observe when the failed login attempts occurred.

SPL Query
index=* EventCode=4625
| timechart span=1m count

The timeline showed that the failed login attempts occurred within a short period, matching the time when I entered the incorrect passwords.

Findings

During the investigation, I confirmed that:

Windows successfully generated Security Event ID 4625 for every failed login attempt.
A total of 11 failed login events were collected by Splunk.
The failure reason for all events was Unknown user name or bad password.
The logon type was Type 2 (Interactive), indicating local authentication attempts.
The source network address was 127.0.0.1 because the failed logins were generated directly on the Windows virtual machine.
All events were successfully forwarded to Splunk through the Universal Forwarder.
What I Learned

This investigation helped me understand how Windows records failed authentication events and how Splunk can be used to analyze Windows Security logs.

I also learned how to use SPL queries to search for failed login events, identify the reason for authentication failures, review important event fields, and visualize login activity over time. These are common tasks performed by SOC analysts when investigating authentication-related alerts.

Conclusion

This investigation demonstrated how Windows Security logs and Splunk can be used together to detect and investigate failed login attempts.

By using simple SPL queries, I was able to identify failed authentication events, review the failure reason, examine the logon type, and create a timeline of the activity. This project improved my understanding of Windows authentication logs and provided hands-on experience with basic SOC investigation techniques.
