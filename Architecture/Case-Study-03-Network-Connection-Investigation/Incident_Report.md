# Case Study 8: Network Connection Investigation

## Objective

The purpose of this investigation was to monitor outbound network connections using Sysmon and Splunk. The goal was to understand how network connection events are recorded, identify which process created the connection, determine the destination IP address and port, and practice investigating network activity as a SOC analyst.

---

# Lab Environment

- **Host Machine:** Windows 10
- **SIEM:** Splunk Enterprise
- **Log Forwarder:** Splunk Universal Forwarder
- **Monitoring Tool:** Sysmon
- **Target System:** Windows 10 Virtual Machine

---

# Scenario

During this lab, I generated outbound network traffic by executing PowerShell commands that sent web requests to Google, GitHub, and Microsoft.

These actions generated **Sysmon Event ID 3 (Network Connection)**, which was forwarded to Splunk through the Universal Forwarder.

My task was to investigate these events, identify the process responsible for the network connections, and review the destination IP addresses and ports.

---

# Investigation Steps

## Step 1 – Search for Network Connection Events

I first searched for all network connection events generated by Sysmon.

### SPL Query

```spl
index=* EventCode=3
```

This displayed every network connection event collected during the investigation.

---

## Step 2 – Filter PowerShell Network Connections

Next, I filtered the results to display only the network connections created by PowerShell.

### SPL Query

```spl
index=* EventCode=3 Image="*powershell.exe"
```

This confirmed that PowerShell generated the outbound network connections during the lab.

---

## Step 3 – Review Network Connection Details

To review the important information, I displayed the key fields from each network connection event.

### SPL Query

```spl
index=* EventCode=3 Image="*powershell.exe"
| table _time Image DestinationIp DestinationHostname DestinationPort Protocol User
```

This allowed me to identify:

- The time of each connection
- The process responsible for the connection
- The destination IP address
- The destination hostname
- The destination port
- The protocol used
- The user who initiated the connection

---

## Step 4 – Count PowerShell Network Connections

I counted the total number of network connections created by PowerShell.

### SPL Query

```spl
index=* EventCode=3 Image="*powershell.exe"
| stats count as "PowerShell Connections"
```

This confirmed that the PowerShell commands successfully generated network connection events.

---

## Step 5 – Identify Destination IP Addresses

Next, I grouped the events by destination IP address.

### SPL Query

```spl
index=* EventCode=3 Image="*powershell.exe"
| stats count by DestinationIp
```

This showed the remote IP addresses contacted by PowerShell during the investigation.

---

## Step 6 – Identify Destination Ports

Finally, I reviewed the destination ports used by the network connections.

### SPL Query

```spl
index=* EventCode=3 Image="*powershell.exe"
| stats count by DestinationPort
```

The results showed that PowerShell primarily used **TCP Port 443 (HTTPS)** to communicate with the remote servers.

---

# Findings

During the investigation, I confirmed that:

- Sysmon successfully generated **Event ID 3** for outbound network connections.
- PowerShell was responsible for creating the network connections during the lab.
- The destination IP addresses of the remote servers were successfully recorded.
- The connections primarily used **TCP Port 443 (HTTPS)**.
- All network connection events were successfully forwarded to Splunk through the Universal Forwarder.

---
